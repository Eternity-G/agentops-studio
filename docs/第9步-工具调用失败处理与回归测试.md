# 第 9 步：工具调用失败处理与回归测试

## 目标

第 9 步的目标是让工具调用失败成为系统可观察、可测试、可回归的状态，而不是静默成功或直接崩溃。

当前覆盖两类失败：

- 计划步骤请求了未注册工具。
- 工具执行过程中抛出异常。

## 核心设计

### ToolResult 增加 error 字段

`ToolResult` 现在包含：

```text
tool_name
status
output
error
```

其中 `error` 用于保存失败原因。成功时为 `None`，失败时保存可解释错误。

### StepExecutionResult 增加 error 字段

`AgentRuntime` 会把 `ToolResult.error` 复制到 `StepExecutionResult.error`。这样 API 响应和 AgentState 都能看到失败原因。

### ToolRegistry 捕获失败

`ToolRegistry.execute_step()` 现在会处理：

| 失败类型 | 行为 |
|---|---|
| 未知工具 | 返回 `FAILED ToolResult`，error 包含 `unknown tool`。 |
| 工具抛异常 | 返回 `FAILED ToolResult`，error 包含 `tool execution failed`。 |

### AgentRuntime 处理失败

当某个步骤工具调用失败时：

1. `step_results` 中保留失败结果。
2. `trace_events` 中记录失败工具事件。
3. `reflect` 阶段发现该步骤没有完成。
4. 整体 `AgentState.status` 变为 `failed`。
5. `errors` 中保存反思阶段发现的问题。
6. `final_answer` 不生成。

这让失败可以被测试、API 和后续评测系统观察到。

## 面试问题

| 问题 | 回答要点 |
|---|---|
| 为什么工具失败不能直接抛到最外层？ | Agent 需要保留中间状态、失败步骤和 trace，便于调试和评测。 |
| 为什么 ToolResult 要有 `error` 字段？ | 结构化失败原因比散落在日志里更容易被 API、Trace、Eval 消费。 |
| 未知工具和工具异常有什么区别？ | 未知工具是路由失败；工具异常是执行失败。二者都应结构化记录。 |
| 为什么 reflect 阶段要检查 completed step？ | 工具失败后不能生成正常最终答案，reflect 能阻止静默成功。 |
| 回归测试有什么价值？ | 工具层会不断扩展，回归测试能防止失败边界被后续改动破坏。 |

## 完成标准

- `ToolResult` 支持 `error`。
- `StepExecutionResult` 支持 `error`。
- 未知工具返回 `FAILED` 结果。
- 工具异常返回 `FAILED` 结果。
- AgentRuntime 遇到失败工具调用时整体状态为 `failed`。
- Trace 中能看到工具失败事件。
- `python -m pytest` 通过。
