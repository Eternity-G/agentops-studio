# 第 6 步：状态机主链路

## 目标

第 6 步的目标是把前面已经做好的模块串成一条最小 Agent 主流程：

```text
任务输入 -> plan -> act -> reflect -> finalize -> 最终状态
```

当前仍然不做真实工具调用。`act` 阶段使用 mock executor，为每个计划步骤生成一个可观测的执行结果。

## 当前状态机

当前主链路包含 4 个阶段：

| 阶段 | 当前做什么 | 后续会扩展什么 |
|---|---|---|
| `plan` | 调用 `Planner` 生成 `ExecutionPlan`。 | 接入真实模型 provider、prompt、结构化输出修复。 |
| `act` | 使用 mock executor 为每个步骤生成完成结果。 | 接入工具调用、SQL、Python、RAG、MCP。 |
| `reflect` | 检查每个计划步骤是否都有完成结果。 | 增加失败分类、重试、重规划。 |
| `finalize` | 生成占位 `FinalAnswer`。 | 基于执行结果生成真实答案和证据引用。 |

## 当前数据结构

| 数据结构 | 作用 |
|---|---|
| `AgentRunStatus` | 表示一次 Agent run 的生命周期状态。 |
| `StepExecutionResult` | 表示一个计划步骤的执行结果。 |
| `AgentState` | 保存任务输入、计划、步骤结果、最终答案和错误信息。 |
| `AgentRuntime` | 负责依次执行 `plan -> act -> reflect -> finalize`。 |

## 面试问题

| 问题 | 回答要点 |
|---|---|
| Agent 状态机解决什么问题？ | 把多步任务拆成明确阶段，让状态、错误和扩展点可观察、可测试。 |
| 为什么现在不用 while loop 一直跑？ | 当前阶段先固定主链路，降低复杂度；后续需要重试和重规划时再引入循环或 LangGraph。 |
| `act` 阶段为什么先用 mock executor？ | 先验证状态流和数据契约，再接真实工具，避免同时调试太多变量。 |
| 为什么需要 `AgentState`？ | 它是跨阶段传递的统一状态，后续 Trace、恢复、评测都依赖它。 |
| `reflect` 阶段现在为什么这么简单？ | 当前只检查步骤覆盖率；后续会扩展为失败分类、证据检查和重规划。 |
| 第 6 步和第 5 步有什么区别？ | 第 5 步只生成计划；第 6 步把计划放进完整 Agent 生命周期。 |
