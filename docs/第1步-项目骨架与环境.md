# 第 1 步：项目骨架与环境

## 目标

第 1 步的目标是搭建一个可以继续扩展的 Python 项目骨架。这个阶段不实现 API、不接入模型、不写 Agent 主链路，只验证项目能被导入、能读取配置、能运行测试。

## 新增文件

```text
app/
  __init__.py
  settings.py
tests/
  test_project_import.py
  test_settings.py
.env.example
.gitignore
pyproject.toml
README.md
```

## 文件职责

| 文件 | 作用 |
|---|---|
| `app/__init__.py` | Python 包入口，暴露项目版本号。 |
| `app/settings.py` | 配置加载模块，集中读取环境变量。 |
| `tests/test_project_import.py` | 验证 `app` 包可以被导入。 |
| `tests/test_settings.py` | 验证配置默认值和环境变量覆盖逻辑。 |
| `.env.example` | 环境变量模板。 |
| `.gitignore` | 忽略虚拟环境、缓存、本地密钥和构建产物。 |
| `pyproject.toml` | 项目元数据、依赖、pytest、ruff、mypy 配置。 |

## 核心设计

### 为什么先做项目骨架

后续 Agent 系统会逐步加入 API、模型适配层、Planner、工具调用、Trace 和 Eval。如果一开始没有清晰的项目结构，后续代码会很快变成脚本堆叠。

第 1 步先建立三个基础约束：

- 所有业务代码放在 `app/`。
- 所有测试放在 `tests/`。
- 配置统一通过 `app/settings.py` 读取。

### 为什么配置不直接读环境变量

如果每个模块都直接调用 `os.getenv()`，后续测试会很难控制配置来源。集中在 `load_settings()` 中读取环境变量，可以让 API、Provider、Eval 使用同一个类型化配置对象。

## 可观测验证

验证项目可以导入：

```bash
python -c "import app; print(app.__version__)"
```

期望输出：

```text
0.1.0
```

验证配置加载：

```bash
python -c "from app.settings import load_settings; print(load_settings())"
```

期望输出类似：

```text
AppSettings(app_name='AgentOps Studio', app_env='development', debug=True, model_provider='mock', default_model='mock-planner-v1')
```

运行测试：

```bash
python -m pytest
```

第 1 步完成时至少应通过 3 个测试。

## 面试问题

| 问题 | 回答要点 |
|---|---|
| 为什么要用 `pyproject.toml`？ | 它是现代 Python 项目的统一配置入口，可以管理构建、依赖、测试和工具配置。 |
| 为什么配置要集中管理？ | 便于测试、替换环境、避免不同模块读取到不一致配置。 |
| 为什么要先写测试？ | 项目每一步都需要可观测验证，避免后续功能堆上去后不知道哪里坏了。 |
| 为什么 `.env` 不能提交？ | `.env` 可能包含 API key、数据库密码等本地敏感信息。 |

## 完成标准

- `python -m pytest` 通过。
- `import app` 正常。
- `load_settings()` 返回类型化配置对象。
- `.env.example` 存在，`.env` 被 `.gitignore` 忽略。
