# 第 15 步：记忆与会话状态

## 本步骤目标

第 15 步实现一个进程内的会话记忆系统，用来保存一次学习或演示会话中的关键事件。

当前记忆系统可以记录：

- Agent 任务运行结果。
- Markdown 文档问答结果。
- 用户手动追加的备注。

注意：本步骤是 **内存记忆**，不是数据库持久化。服务重启后数据会丢失。

## 为什么先做内存记忆

记忆系统很容易变复杂：数据库、用户权限、长期摘要、向量记忆、隐私删除、过期策略都会影响设计。

第 15 步先做最小版本，目标是把工程边界跑通：

```text
session_id -> memory events -> query session state
```

这样后续接 SQLite、PostgreSQL 或 Redis 时，上层 API 和测试可以尽量保持稳定。

## 核心设计

### 1. SessionState

`SessionState` 是一个会话的当前内存状态：

- `session_id`：会话 id。
- `events`：按顺序追加的事件列表。

### 2. MemoryEvent

每条记忆都是一个事件：

- `event_id`：事件唯一 id。
- `event_type`：事件类型。
- `summary`：短摘要，方便快速浏览。
- `payload`：结构化调试数据。
- `created_at`：创建时间。

当前支持三类事件：

- `task_run`
- `document_qa`
- `note`

### 3. 写入时机

`POST /tasks/run` 支持通过 `metadata.session_id` 写入记忆：

```json
{
  "task": "分析下一步计划",
  "metadata": {
    "session_id": "demo-session"
  }
}
```

`POST /documents/ask` 支持通过 `session_id` 写入记忆：

```json
{
  "question": "第 14 步是什么？",
  "path": "docs/第14步-文档问答最小RAG链路.md",
  "session_id": "demo-session"
}
```

手动备注通过独立接口写入：

```text
POST /sessions/{session_id}/notes
```

## API

查询会话：

```bash
curl http://localhost:8000/sessions/demo-session
```

追加备注：

```bash
curl -X POST http://localhost:8000/sessions/demo-session/notes ^
  -H "Content-Type: application/json" ^
  -d "{\"content\":\"这是一次学习备注\",\"metadata\":{\"source\":\"manual\"}}"
```

## 你需要理解的面试问题

### 1. 内存记忆和长期记忆有什么区别？

内存记忆只存在于当前 Python 进程里，重启后消失，适合 Demo、测试和本地学习。

长期记忆需要持久化到数据库或外部存储，并考虑权限、删除、迁移、备份和容量。

### 2. 为什么记忆使用事件列表，而不是只存一个最终状态？

事件列表保留了过程。

Agent 系统需要知道发生过什么：任务什么时候运行、文档问答用了哪个来源、人工备注是什么。事件流比单个最终状态更适合 trace、评测和调试。

### 3. 为什么 payload 只保存摘要信息？

因为记忆不是日志归档。

如果把完整计划、完整文档和完整 trace 都塞进内存，很快会失控。当前只保存调试和演示需要的关键字段。
