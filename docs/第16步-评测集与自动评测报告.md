# 第 16 步：评测集与自动评测报告

## 本步骤目标

第 16 步建立一个离线、可重复运行的最小评测体系。

评测覆盖当前已经实现的三条核心链路：

- Agent 任务主链路。
- Markdown 文档问答链路。
- 内存会话记忆链路。

本步骤不调用真实 DeepSeek API，默认使用 `MockProvider`，保证评测稳定、低成本、适合 CI。

## 为什么需要评测

Agent 项目不能只靠“接口能跑”证明质量。

评测系统的作用是：

- 固定核心用例，防止后续改动破坏主链路。
- 把质量验证变成可重复命令。
- 生成 Markdown 报告，方便展示和复盘。
- 为后续简历项目提供工程亮点。

## 评测集设计

当前 `evals/cases.json` 包含 3 类用例。

### 1. task_run

验证 Agent 主链路可以完成：

```text
task -> plan -> act -> reflect -> finalize
```

### 2. document_qa

验证文档问答链路可以读取本地 Markdown，并返回包含期望内容的答案。

### 3. session_memory

验证会话记忆可以保存人工备注事件。

## 评测报告

运行命令：

```bash
python scripts/run_evals.py
```

脚本会：

1. 读取 `evals/cases.json`。
2. 使用 `EvaluationRunner` 执行评测。
3. 输出 Markdown 报告。
4. 写入 `evals/latest-report.md`。
5. 如果存在失败用例，进程返回非 0 状态码。

报告示例：

```text
通过率：100%
```

## 为什么评测默认使用 MockProvider

评测必须稳定。

真实模型会受到 API key、网络、限流、价格和输出随机性的影响。如果离线评测默认依赖真实模型，测试结果就很难复现。

所以第 16 步的 `EvaluationRunner` 默认使用 `MockProvider`。

真实模型质量评测可以在后续单独设计，例如：

- 手动触发。
- 只在有 API key 时运行。
- 记录模型名和成本。
- 区分稳定性评测和效果评测。

## 你需要理解的面试问题

### 1. Agent 项目为什么需要评测？

因为 Agent 的行为链路长，失败点多。只看一次 Demo 成功不够，必须有固定用例持续验证核心能力。

### 2. 单元测试和评测有什么区别？

单元测试更关注代码边界是否正确。

评测更关注系统行为是否达到预期，例如任务是否完成、答案是否包含关键证据、会话是否保存关键事件。

### 3. 为什么失败时脚本要返回非 0？

因为这样才能接入 CI。

如果评测失败但脚本仍返回 0，自动化系统就无法识别质量退化。
