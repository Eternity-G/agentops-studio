# 简历项目描述

## 项目名称

AgentOps Studio：面向 Agent 工程学习与演示的全链路平台

## 一句话描述

从零构建一个包含任务规划、模型适配、工具调用、Trace、文档问答、会话记忆、自动评测和前端 Demo 的 Agent 工程平台。

## 简历写法

可以写成：

```text
AgentOps Studio | Python / FastAPI / Pydantic / LLM Provider / Agent Workflow

- 从零设计并实现 Agent 工程平台，覆盖任务输入、结构化规划、工具调用、状态机执行、Trace、文档问答、会话记忆和自动评测。
- 基于 Pydantic 设计核心数据契约，包括 TaskInput、ExecutionPlan、PlanStep、FinalAnswer、DocumentQuestionAnswer 和 SessionState，保证模型输出和 API 响应可校验。
- 实现统一 ModelProvider 接口，支持 MockProvider 与 DeepSeek OpenAI-compatible Provider，并为真实模型输出增加重试、schema 再校验和 fallback 机制。
- 设计工具层与 ToolRegistry，接入本地 Markdown 读取工具，限制读取范围和输出长度，为最小 RAG 链路提供安全边界。
- 使用 FastAPI 暴露健康检查、任务运行、文档问答、会话记忆和评测报告接口，并提供静态前端 Demo 页面。
- 建立离线 EvaluationRunner 和固定评测集，自动生成 Markdown 评测报告，覆盖 Agent 主链路、文档问答和会话记忆。
```

## 面试讲解主线

建议按这条线讲：

1. 为什么要做 AgentOps Studio，而不是普通聊天机器人。
2. 如何用 Pydantic 把不稳定的 LLM 输出约束成稳定 schema。
3. 如何抽象 Provider，使 MockProvider 和 DeepSeekProvider 可以替换。
4. 如何处理真实模型失败：重试、再校验、fallback、trace。
5. 如何设计工具层：ToolDefinition、ToolResult、ToolRegistry。
6. 如何从 Markdown 工具演进到最小 RAG 链路。
7. 为什么需要会话记忆和事件流。
8. 为什么需要离线评测，而不是只靠手动 Demo。
9. 前端 Demo 和 Docker 部署如何证明项目可交付。

## 可展示亮点

- 有明确的 18 步工程路线。
- 每一步都有测试和文档。
- 支持真实 DeepSeek API，但单元测试不依赖真实网络。
- 有 fallback 和 trace，不把模型失败隐藏掉。
- 有最小 RAG、会话记忆和评测报告。
- 有浏览器 Demo 和 Dockerfile。

## 当前边界

这个项目目前仍是学习型平台，不是生产级商业系统。

还可以继续扩展：

- 持久化数据库。
- 向量数据库和 embedding 检索。
- 用户认证和权限。
- 更完整的前端工程。
- 云端 CI/CD。
- 真实工具调用和人工审批。
