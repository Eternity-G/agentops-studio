# 项目复盘

## 项目定位

AgentOps Studio 是一个 Agent 工程学习项目。

它解决的问题不是“如何调用一次大模型”，而是“如何把大模型调用放进一个可测试、可观测、可扩展的工程系统里”。

## 已完成模块

| 模块 | 当前能力 |
|---|---|
| API 服务 | FastAPI 应用、健康检查、OpenAPI 文档。 |
| Schema | 任务、计划、步骤、答案、文档问答、会话记忆和评测数据契约。 |
| Provider | MockProvider、DeepSeekProvider、Provider 工厂。 |
| Planner | 任务输入到结构化 ExecutionPlan。 |
| 失败恢复 | provider 重试、输出再校验、fallback、trace 记录。 |
| Agent Runtime | `plan -> act -> reflect -> finalize` 最小状态机。 |
| 工具层 | Tool 协议、ToolRegistry、Mock 工具、Markdown 读取工具。 |
| 文档问答 | 本地 Markdown 读取、片段抽取、带证据回答。 |
| 会话记忆 | 任务、文档问答和备注事件写入内存 session。 |
| 评测 | 固定评测集、离线 runner、Markdown 报告。 |
| 前端 | FastAPI 托管静态 Demo 页面。 |
| 部署 | Dockerfile、本地部署说明、验收命令。 |

## 关键工程决策

### 1. 先 Mock，再接真实模型

先实现 MockProvider，可以让主链路和测试不依赖真实 API key。

等 schema、planner、runtime 都稳定后，再接 DeepSeekProvider。

### 2. schema 是系统边界

Pydantic schema 是这个项目的稳定契约。

它让模型输出、API 请求、工具结果和评测结果都有明确结构，降低后续扩展风险。

### 3. 失败要可观测

真实模型会失败，工具也会失败。

项目没有把失败隐藏为成功，而是通过 `ToolResult`、`AgentState.errors` 和 `trace_events` 暴露出来。

### 4. 评测独立于真实模型

离线评测默认使用 MockProvider。

这样评测结果稳定、可重复、低成本，不受网络和 API key 影响。

## 当前限制

- 记忆仍是进程内存储，服务重启后丢失。
- 文档问答还没有 embedding 和向量检索。
- 前端是 Demo 页面，不是完整前端产品。
- DeepSeek 真实模型调用还没有成本统计和限流策略。
- 没有用户认证、权限和多租户隔离。

## 后续增强方向

1. 接 SQLite 或 PostgreSQL 保存会话和 trace。
2. 引入向量数据库，实现真正 RAG。
3. 增加真实外部工具，例如搜索、文件分析、代码执行沙箱。
4. 引入 LangGraph 或自研图状态机。
5. 增加 CI/CD 和云端部署。
6. 增加前端构建工程和用户体验细节。
